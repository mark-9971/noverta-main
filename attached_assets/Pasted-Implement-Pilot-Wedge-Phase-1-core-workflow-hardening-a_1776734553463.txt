Implement Pilot Wedge Phase 1 — core workflow hardening and nav simplification.

Use the main agent. Do the work, not just a plan.

Goal:
Make the pilot-critical daily workflow feel tight, credible, and easy to use for a district admin / coordinator, without broad redesign or random feature sprawl.

Primary product objective:
A user should be able to move through the core loop quickly:

1. Know who is at risk
2. Understand why
3. See what to do next
4. Prove / review the underlying details

Core pilot wedge surfaces:
- Dashboard
- Compliance
- Action Center
- Student detail
- Today
- Quick log / session logging
- scheduling visibility where it directly supports the above loop

Context:
The School Calendar v0 work is now strong enough:
- closures
- early release
- downstream read math
- forecast/timing-aware paths
- consequence alignment
- weekday-aware baseline

Do NOT keep deepening calendar work unless you find a direct blocker in one of the core wedge surfaces. The current goal is product usability and coherence, not more subsystem nuance.

What I want you to do:
Audit the current pilot-critical experience as a product architect + staff engineer, then implement the highest-value improvements that make the core wedge more usable and less confusing.

Focus areas:
A. Navigation / information architecture
- identify cyclical, duplicative, or confusing entry points across Dashboard, Compliance, Action Center, Student detail, Today, and session logging
- reduce unnecessary overlap or redundant paths where possible
- improve the “next obvious click” from one core surface to another
- hide, de-emphasize, or stop routing users toward non-core distractions if that can be done safely without broad redesign

B. Core workflow clarity
Improve the flow from:
- Dashboard → why is this student/school at risk?
- Compliance → what specifically is wrong?
- Student detail → what action should we take?
- Today / session logging → how do providers actually close the gap?

C. Student detail page usefulness
Student detail is likely part of the pilot-critical loop. Improve it if needed, especially if:
- the page is too cluttered
- the most important compliance/service info is buried
- actions are too hard to find
- tab structure or layout hurts core use

D. Actionability
Where the product currently reports risk but does not guide action well, improve the path to action without inventing a giant new workflow.
Examples:
- better links to relevant student detail or logging surfaces
- clearer “why this matters” summaries
- less dead-end reporting
- better placement of actions already available in the codebase

E. Keep this phase bounded
This is NOT a broad redesign and NOT a visual beautification project.
Do not:
- build a whole new scheduling system
- redesign the whole product shell
- add speculative new modules
- reopen deep calendar architecture unless a core wedge bug requires it
- spread across unrelated non-pilot surfaces

Execution strategy:
1. Audit the pilot wedge first
2. Identify the highest-value usability/flow issues
3. Implement the improvements with the best ROI
4. Prefer focused changes in shared navigation, page composition, and action flow over giant rewrites
5. If one or two surfaces are especially messy (for example student detail), it is okay to structurally clean them up if it directly improves the wedge

What I care about most:
- fewer confusing paths
- better coherence between pages
- better explanation of risk
- better path from insight to action
- less nav clutter around the pilot-critical loop
- preserving correctness and trust
