Implement Pilot Wedge Phase 1C — student detail action surface, compliance/risk row actionability, return-to-queue workflow closure, and server-side handling state if feasible.

Use the main agent. Do the work, not just a plan.

Goal:
Finish the highest-value remaining Pilot Wedge Phase 1 work in one coherent pass by extending the operational action model beyond Action Center into the other core wedge surfaces that still feel passive or dead-end, while also attempting to replace local-only handling state with a real shared server-side model if that can be done cleanly.

Primary product objective:
A district admin / coordinator should be able to move through the core loop with minimal friction:

1. identify risk
2. understand likely cause
3. see the single recommended next action
4. take or hand off that action
5. see whether it is already being handled
6. return to the queue and continue with the next student / issue

Phase 1A is complete:
- Action Center always opens Quick Log inline
- dashboard variants have a clearer Action Center funnel
- pilot/focused scheduling nav was simplified

Phase 1B is partially complete:
- Action Center now uses a centralized cause-aware / owner-aware recommendation model
- handling state exists but is currently localStorage-backed
- “Log Session” is no longer the universal default

This phase should finish the biggest remaining wedge gaps, not start a new product area.

==================================================
A. WHAT THIS PHASE MUST DELIVER
==================================================

There are five target deliverables in this phase:

1. Student detail becomes a real action surface
2. Compliance / risk row surfaces become actionable using the same model
3. Return-to-queue behavior is explicit and reliable
4. The new action model is reused, not reinvented
5. Server-side/shared handling state is implemented if it can be done cleanly without derailing the phase

If #1–#4 are not completed in a meaningful way, verdict should be PARTIAL.
If #5 cannot be completed cleanly in the same pass, do NOT fake it — keep the strongest honest implementation and explicitly declare the exact next task required.

==================================================
B. STUDENT DETAIL — REQUIRED OUTCOME
==================================================

Student detail must stop feeling like a passive verification stop and become a clear “what should we do for this student now?” surface.

Implement a high-signal Recommended Next Step surface on the most important student-detail summary area (likely TabSummary or the true equivalent).

That surface must show, for the student’s highest-priority current issue:
- likely cause
- confidence / ambiguity state if relevant
- recommended owner
- single recommended next action
- current handling state if one exists
- one-sentence “why this recommendation” explanation
- a small set of secondary available actions where appropriate

Important:
- This must reuse the centralized recommendation model from the Action Center work
- Do not create a second recommendation engine for student detail
- If the student has multiple issues, choose one clear top-priority recommendation and optionally summarize that more issues exist
- Do not bury the Recommended Next Step below low-priority cards

Good examples:
- “Recommended next step: Schedule makeup”
- “Likely cause: likely missed service”
- “Owner: scheduler”
- “Why: 2 scheduled blocks were missed and no delivery was logged”
- Handling state: “Awaiting confirmation” / “Recovery scheduled” / etc. if present

Student detail also must support queue-aware return behavior:
- if opened from Action Center, show a clear “Back to Action Center” affordance
- if opened from Compliance / risk list, show a clear “Back to Compliance” affordance
- do not rely on browser back button alone
- preserve enough context to continue the queue/list workflow

==================================================
C. COMPLIANCE / RISK SURFACES — REQUIRED OUTCOME
==================================================

The most central compliance/risk list or report row surface must stop being passive.

Choose the highest-ROI row/list surface where admins actually review at-risk students/services in the pilot wedge, and apply the same operational action language there.

At minimum, each important row should have:
- likely cause subline
- recommended owner
- single recommended next action
- a small secondary action affordance if needed
- if applicable, current handling state
- if appropriate, inline Quick Log only when the recommendation model says documentation lag is likely

Important:
- Do not add generic “Log Session” everywhere
- Do not duplicate the recommendation logic
- If the row surface is too dense for full treatment, at least add a compact but meaningful version of the same model
- If multiple compliance/risk surfaces exist, choose the most central/highest-value one and explain why in the output

Examples of acceptable compact row treatment:
- “Likely cause: documentation lag • Owner: provider”
- Primary CTA: “Confirm & log”
- Secondary menu: “Schedule makeup”, “Open student”, etc.

or

- “Likely cause: likely missed service • Owner: scheduler”
- Primary CTA: “Schedule makeup”
- Secondary menu: “Open student”, “Mark handed off”, etc.

==================================================
D. RETURN-TO-QUEUE / THROUGHPUT — REQUIRED OUTCOME
==================================================

This is a hard requirement.

After an admin/coordinator takes the recommended action or initiates a handoff, the product should return them to the originating operational context when appropriate so they can work the next issue immediately.

That means:

- If acting from Action Center:
  - stay in Action Center whenever possible
  - preserve tab/filter/scroll/list context
  - update the row in place if possible
  - make it easy to continue to the next unresolved item

- If drilling into student detail from Action Center:
  - there must be a visible “Back to Action Center” affordance
  - recommended-next-step actions taken there should not strand the user

- If drilling into student detail from Compliance/risk:
  - there must be a visible “Back to Compliance” affordance
  - preserve enough context to continue that queue/list

Prefer:
- inline sheets/drawers
- query-param/context preservation
- status refresh in place
- keeping the user in the originating queue/list

Avoid:
- forcing deep navigation after every action
- making the user re-find their place
- relying only on browser back

This is not just navigation polish; it is queue-based operational throughput.

F. SERVER-SIDE HANDLING STATE — STRONGLY PREFERRED
==================================================

The current localStorage-only handling state is not sufficient for a real multi-user school workflow.

In this phase, attempt to promote handling state to a real server-side/shared model if it can be done cleanly and safely without derailing the phase.

At minimum, a server-side handling model should support:
- stable item id
- current handling state
- current/recommended owner if applicable
- updated_by
- updated_at

If you can implement this cleanly in the same pass, do it and wire the touched pilot-wedge surfaces to use it.

If you cannot do it cleanly without broadening scope too much, then:
- do NOT fake cross-user/shared behavior
- keep the strongest honest bounded implementation
- explicitly state that server-side handling remains the next highest-priority task
- list the exact schema/routes/components needed to finish it next

Important:
Do not leave the final output vague on this point. State clearly whether handling state is:
- fully server-side/shared
- partially server-side/shared
- still local-only by design for now

If you do implement server-side handling:
- prefer reusing existing stable work-item identifiers if available
- if a new minimal persistence model is necessary, keep it narrowly scoped to this operational handling purpose
- do not turn this into a giant task/notification platform
- touched surfaces must read/write the shared state consistently

If you do not implement server-side handling:
- do not let the local-only approach block #1–#4
- but be explicit and concrete about the remaining work
