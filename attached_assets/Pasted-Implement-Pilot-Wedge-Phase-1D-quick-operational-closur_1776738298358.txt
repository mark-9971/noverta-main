Implement Pilot Wedge Phase 1D — quick operational closures before shared handling-state infrastructure.

Use the main agent. Do the work, not just a plan.

Goal:
Finish the highest-ROI remaining visible Phase 1 wedge improvements before the bigger server-side handling-state pass. This phase should make the current operational action model feel more executable, more visible on key surfaces, and less duplicative.

Primary product objective:
When an admin/coordinator sees a student/service behind on minutes, the product should not just recommend the right next step — it should make that step easier to initiate from the most important surfaces, and it should reduce duplicate chasing in the visible wedge.

Context:
Phase 1A is complete:
- Action Center opens Quick Log inline
- dashboard has clearer Action Center funnel
- focused nav is trimmed

Phase 1B is complete:
- centralized action recommendation model exists
- Action Center shows likely cause, owner, recommended next action
- local handling state exists

Phase 1C is complete for visible wedge work:
- student detail has a Recommended Next Step card
- compliance Risk Report has recommendation subline
- return-to-queue via ?from=action-center / ?from=compliance is implemented
- server-side handling state was explicitly deferred honestly

The highest-value remaining visible gaps are now:
1. “Schedule makeup” is still mostly a recommendation, not a real launch path
2. Risk Report rows still require one extra click to act
3. owner-aware “you” wording is incomplete
4. dashboard/needs-attention surfaces do not yet reflect handling-state enough to reduce duplicate chasing
5. HANDLING_TRANSITIONS is duplicated and should be hoisted while context is fresh

==================================================
B. SCHEDULE MAKEUP — REQUIRED OUTCOME
==================================================

This is the highest-priority visible fix.

Right now “Schedule makeup” is often just a recommendation / state update. That is not enough.
Make it launch a real operational path.

Required behavior:
- when the recommendation is schedule_makeup, the primary CTA should route/open the scheduling surface with as much prefilled context as safely possible
- pass through at minimum:
  - studentId
  - serviceRequirementId if available
  - missedSessionId or best available shortfall context if available
  - intent=makeup
- do not invent a giant new scheduling system
- reuse the existing scheduling surface and query-param patterns if possible
- preserve the queue-oriented workflow as much as practical

Preferred behavior:
- if launched from Action Center or student detail, the user should clearly be in a “makeup intent” mode
- if returning is already supported by query params or existing context patterns, preserve it
- if a perfect round-trip cannot be completed in this phase, still make the launch operationally useful and report the exact remaining gap honestly

==================================================
C. RISK REPORT INLINE ACTIONABILITY — REQUIRED OUTCOME
==================================================

The compliance Risk Report should stop being one click more passive than necessary.

Implement a stronger row treatment for the highest-value at-risk rows (for example out_of_compliance / highest-priority rows) so the row itself is more actionable.

At minimum, for the chosen row treatment:
- show the recommended next action clearly
- show owner clearly
- show handling-state pill
- offer the primary CTA inline where density allows
- keep the row readable and not cluttered
- continue to allow click-through to student detail

Important:
- reuse the centralized recommendation model
- do not duplicate logic
- do not make “Log Session” generic
- if row density forces a compact pattern, prioritize the highest-value row state(s) rather than trying to treat every row equally

==================================================
D. OWNER-AWARE “YOU” SUBSTITUTION — REQUIRED OUTCOME
==================================================

Extend the current owner-aware wording beyond the narrow existing provider case.

Where the user’s current role clearly matches the recommended owner, the UI should say “you” instead of a generic role label in the relevant touched surfaces.

Target at minimum:
- service provider
- case manager / SPED teacher if role mapping exists cleanly
- scheduler if role mapping exists cleanly
- admin when appropriate

Do this only where it is actually justified by the current role/context.
Do not fake owner identity if the system cannot infer it reliably.

==================================================
E. DASHBOARD / NEEDS-ATTENTION HANDLING VISIBILITY — REQUIRED OUTCOME
==================================================

The dashboard should do a better job of not making already-in-progress items look identical to untouched items.

Improve the highest-value dashboard / NeedsAttention surface so that handling-state affects presentation enough to reduce duplicate chasing.

Acceptable implementations include:
- grey out items already in non-default handling states
- exclude items already marked in-progress from “needs action” counts if safe
- separate “needs action” from “already in progress”
- show a lightweight in-progress label/state

Important:
- do not break trust in counts
- do not create a misleading “resolved” illusion if the state is still only local-only
- keep it honest and useful

Move HANDLING_TRANSITIONS to a single shared source of truth so Action Center and RecommendedNextStepCard no longer duplicate the constant.
