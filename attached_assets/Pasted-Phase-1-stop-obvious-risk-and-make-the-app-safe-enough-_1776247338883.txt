Phase 1 — stop obvious risk and make the app safe enough to keep building

These come first because they remove avoidable landmines.

1. Remove or fully gate the role switcher

Goal: ensure no user can elevate or simulate roles outside development.

Do:
	•	hide role switcher entirely in production
	•	enforce NODE_ENV === "development" check server-side too if relevant
	•	verify no client-side role state can affect server authorization
	•	search codebase for any other dev-only shortcuts

Done when:
	•	production build has no visible role switcher
	•	changing client state cannot change API access

2. Switch to production auth configuration and environment separation

Goal: stop using dev auth posture.

Do:
	•	set up separate dev / staging / prod Clerk environments
	•	rotate keys
	•	verify callback URLs, allowed origins, and environment configs
	•	audit all env vars and secrets usage

Done when:
	•	no dev key warnings anywhere
	•	staging and prod are independently configured

3. Enforce tenant isolation at the API layer

Goal: every read/write must be scoped to the user’s org/school/district claims.

Do:
	•	define canonical tenant context from auth
	•	require school/district/org scoping in middleware
	•	remove trust in client-passed filter params
	•	audit all routes for cross-tenant access risk

Done when:
	•	a user cannot query another org’s records even with manipulated requests

4. Add API rate limiting

Goal: basic abuse protection, especially on exports/auth-sensitive endpoints.

Do:
	•	add global rate limiting
	•	stricter limits on exports, search, login-adjacent, and report generation endpoints
	•	add audit logging for limit hits

Done when:
	•	bulk scraping/export attempts are slowed or blocked

5. Create a permission test matrix for critical roles

Goal: verify auth and role boundaries before more feature work.

Test at minimum:
	•	admin
	•	case_manager
	•	bcba
	•	provider
	•	para
	•	student

Cover:
	•	students
	•	sessions
	•	incidents
	•	reports
	•	exports
	•	district/admin pages

Done when:
	•	automated tests prove forbidden actions fail correctly

Phase 2 — make outputs real

This is the biggest product gap.

6. Build a shared document rendering system

Goal: create one reusable document/output pipeline, not one-off hacks.

Choose a path:
	•	HTML print templates + print CSS for fast iteration
	•	then optionally Puppeteer / React-PDF for durable exports

Support:
	•	consistent headers
	•	student metadata
	•	dates
	•	signature blocks
	•	print-friendly layout
	•	PDF download

Done when:
	•	one document type can render consistently in browser and PDF

7. Ship incident / protective-measures printable report

Goal: close the loop on your strongest compliance workflow first.

Include:
	•	incident details
	•	staff involved
	•	injuries/medical attention
	•	review status
	•	notification history
	•	DESE-related fields
	•	signature/ack lines

Done when:
	•	an admin can generate and print/download a complete incident packet

8. Ship progress report printable/PDF output

Goal: turn narrative generation into usable output.

Include:
	•	student info
	•	reporting period
	•	goal-by-goal progress
	•	narrative sections
	•	related services summary if applicable

Done when:
	•	a teacher can generate a progress report and actually send/file it

9. Ship IEP Builder print/download output

Goal: make your most differentiated feature materially useful.

Include:
	•	structured sections from wizard
	•	recommended goal language
	•	rationale / data summary
	•	editable review mode before export

Important:
This does not have to be a legally final IEP document on day one. It can be:
	•	draft packet
	•	team prep document
	•	draft recommendation set

Done when:
	•	user can generate, review, and download a usable artifact

10. Add a “document status” model

Goal: track draft/final/archived, rather than random generated text blobs.

For outputs like:
	•	progress reports
	•	incident reports
	•	IEP drafts
	•	evaluation summaries later

Done when:
	•	generated documents can be revisited, versioned, and re-downloaded

Phase 3 — make communication real

Logging fake sends is dangerous. Fix that next.

11. Add real email delivery integration

Goal: actual sending with delivery status.

Use:
	•	Resend, SendGrid, or SES

Support:
	•	parent notification emails
	•	alert emails
	•	follow-up reminders
	•	failure states and retries

Done when:
	•	the system sends real emails and records actual delivery attempts

12. Add email event tracking and communication audit trail

Goal: prove what was sent, when, to whom, and whether it failed.

Track:
	•	drafted
	•	queued
	•	sent
	•	delivered if available
	•	failed/bounced
	•	manually printed instead

Done when:
	•	“sent” only appears after a real send event, not a button click

13. Fix protective-measures parent notification workflow end to end

Goal: make this legally safer and operationally real.

Flow:
	•	generate draft
	•	review/edit
	•	send or print
	•	record delivery channel
	•	require confirmation/status update

Done when:
	•	no incident can misleadingly appear “sent” without real delivery evidence

14. Add templated reminder/alert notifications

Goal: turn passive dashboards into workflow-driving software.

Start with:
	•	missed session alert
	•	overdue parent follow-up
	•	incomplete transition plan
	•	overdue evaluation task

Done when:
	•	key alerts can generate targeted reminders

	•	minimal fields
	•	saved defaults
	•	recent students/services

Done when:

Features:

	•	tap completed/missed
	•	optional short note
	•	submit

Done when:

Done when:
