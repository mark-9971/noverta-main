Objective

Complete T04 in the active closed-loop makeup chain:

When a user logs a makeup session against a linked schedule block, the server must:

copy / persist the originating source_action_item_id onto the new session_logs row if it is not already being carried through
resolve the matching shared handling-state item server-side
write the appropriate audit trail
make the wedge reflect the closed loop without requiring a second manual resolve action

This is the next task after T03. T03 is already accepted as complete and now distinguishes deliveredMinutes, scheduledPendingMinutes, and stillAtRiskMinutes, but it explicitly left a bounded seam that T04 must close: until the session log is linked back to the originating action item, a manually logged makeup can briefly remain visible as pending instead of fully resolving.

Scope

Implement the smallest correct server-side closed-loop behavior for linked makeup logging.

In scope
A. Persist the originating action-item link on session log creation

When a session is logged from / against a makeup schedule block that has source_action_item_id:

ensure the newly created session_logs row stores that same source_action_item_id
do this in the canonical server-side logging path
do not rely on the client to supply or preserve this linkage correctly

If the current logging flow already passes a scheduleBlockId or equivalent linkage, use that canonical relationship rather than inventing a new one.

B. Resolve the matching handling-state row server-side

After the session log is successfully written:

find the matching action_item_handling row for that source_action_item_id
transition it to the canonical resolved state on the server
preserve district scoping / tenant safety
do not require the user to separately click resolve in Action Center or another surface
C. Write audit trail for the resolution

The server-side transition must be auditable.

At minimum, the result should make it possible to prove:

which session log caused the resolution
which action item was resolved
who performed the session log action / resolution
when it happened

If a canonical audit fan-out already exists, reuse it.
If the canonical handling-state transition path already records history, call through it rather than duplicating equivalent writes.

D. Preserve correct closed-loop behavior for T03 math

T03 already introduced the no-double-counting model:

delivered = authoritative session log minutes
scheduledPending = linked makeup blocks not yet represented by a session log carrying the same source_action_item_id
stillAtRisk = residual gap

T04 must make that model operational by ensuring the linkage exists at session-create time, so the block drops out of pending and the minutes count once, as delivered.

Workflow expectations
Before

User schedules / has a linked makeup block, then logs the makeup session.
The minutes may become delivered, but the originating handling item can remain pending until someone manually resolves it. The loop is not truly closed.

After

User logs the makeup session against the linked block.
The server:

writes the session log
mirrors the source_action_item_id
resolves the matching handling item
preserves audit/history
allows the wedge to reflect the issue as closed without a second manual step

This is the functional close of the server-side recovery loop. T05 will make it visible consistently across surfaces later.
