## Exact execution order

### Step 1 — inspect and map the real UI flow
Find the actual user-facing entrypoint(s) for:
- Add Sample Data
- Reset District
- Demo Reset
- any equivalent operator controls

You must identify:
- where the user clicks
- what forms/options exist
- what feedback is shown
- whether size controls are visible
- whether the flow is understandable without insider knowledge

### Step 2 — execute one real UI path end-to-end
Using the real UI, perform one end-to-end sample/reset action as a normal user would.

Record:
- entrypoint used
- inputs used
- what happened
- whether it succeeded
- what was shown after completion

### Step 3 — verify resulting state in the product
After the action completes, verify through the UI/product that:
- the district was actually seeded/reset
- data is visible
- seeded district feels usable
- overlay/showcase impact is present where naturally expected
- summary/feedback matches reality as much as current UI exposes it

### Step 4 — identify blockers / UX gaps
Be explicit about:
- missing size controls
- confusing labels
- missing progress feedback
- unclear completion state
- mismatch between backend capability and UI exposure
- any places where a normal demo user would get stuck

### Step 5 — STOP if changes are needed
If the UI flow is incomplete, confusing, or missing required controls:
- do not implement immediately
- return the findings
- ask for permission before changing code

Only proceed to implementation if permission is explicitly granted.
