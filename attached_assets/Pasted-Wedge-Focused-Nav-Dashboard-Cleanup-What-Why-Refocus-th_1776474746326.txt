Wedge-Focused Nav & Dashboard Cleanup
What & Why
Refocus the Trellis product experience around its core wedge — SPED service-delivery, compliance-risk, and compensatory-exposure management — by trimming the primary sidebar to wedge-critical items, demoting secondary and exploratory features into the existing "More tools" collapsed section, and removing non-wedge CTAs from the dashboard. No routes, pages, APIs, DB schema, or business logic are deleted. Every change is nav-level and reversible.

Done looks like
Primary sidebar (admin role) shows only the wedge: Dashboard, Alerts, Compliance section (service minutes / risk / compensatory), a lean Caseload section (Students, Sessions, Schedule, Coverage, Staff), Reports, and Settings
Billing section is collapsed by default so it does not compete with the compliance wedge
Secondary items (IEP Search, Evaluations, IEP Calendar, Staff Calendar, IEP Meetings, Caseload Balancing, Progress Reports) are reachable from the existing "More tools" collapsed section — routes still work when visited directly
Staff Directory is promoted out of "More tools" into the Caseload section so the core provider/staff workflow is one click away
The nav config has a visibility property ('primary' | 'secondary' | 'hidden') on NavItem and NavSection as a centralized, single-place control layer for future adjustments
Dashboard does not surface prominent CTAs for demoted/secondary modules; existing compliance, session-gap, and onboarding cards remain intact with no regressions
sped_teacher/BCBA nav inherits the updated Caseload section correctly (label maps, excluded groups still apply)
App builds clean, no TypeScript errors introduced, all protected routes still resolve, hidden routes still render when visited directly
Out of scope
Deleting any route, page, API endpoint, database table, or business logic
Changing any page content or feature functionality
Changes to the guardian portal or SPED student portal navs
Changes to the para nav
Adding new feature flags or tier gates
Modifying the mobile bottom nav behavior (it already uses primary: true items)
Any schema migrations
