You are acting as a senior QA engineer, product tester, and full-stack auditor for this app. Your goal is to prepare it for a live demo / pilot presentation using my Clerk dev account.

Context:
- This app must feel polished and reliable enough for a demo, even if it is not fully production-ready.
- Prioritize real-world functionality, broken flows, bad UX, missing states, data integrity issues, auth issues, and anything that would embarrass me in a presentation.
- Assume I will log in through my Clerk dev account and demo the app end-to-end.
- Do not just review code statically. Actively inspect the app, run it, click through flows, trace code paths, identify weak points, and verify behavior against expected product intent.
- Be skeptical. Look for hidden failures, half-built features, broken edge cases, console errors, incorrect permissions, and fake or disconnected UI.

Your mission:
Perform a thorough demo-readiness audit of the entire app and produce a structured report plus an action plan.

Audit scope:
1. Authentication / user access
- Verify Clerk auth is correctly implemented throughout the app
- Confirm sign up, sign in, sign out, session persistence, protected routes, redirect behavior, and user state hydration
- Confirm there are no pages or API routes unintentionally exposed without auth
- Confirm my Clerk dev account can access the intended demo data and flows cleanly
- Identify any auth bugs, race conditions, missing guards, or broken loading states
- Check for places where auth exists in UI but is not actually enforced on backend/API

2. Core product flows
- Identify every major feature and user flow in the app
- Test each one end-to-end from UI to backend/database
- Mark each as:
  - Working
  - Partially working
  - Broken
  - Placeholder / fake
- For each feature, describe:
  - expected behavior
  - actual behavior
  - where it breaks
  - severity for demo risk
- Pay special attention to create, edit, delete, save, filtering, search, dashboards, exports, uploads, generated content, form submissions, and state persistence

3. UI / UX demo-readiness
- Find anything visually broken, confusing, unfinished, inconsistent, or embarrassing in a live presentation
- Check for:
  - empty states
  - loading states
  - error states
  - button labeling
  - dead buttons
  - fake stats
  - layout issues
  - mobile/responsive issues if relevant
  - pages that look complete but are not actually functional
- Flag places where I may click something in a demo and get stuck or confused

4. Backend / API / database integrity
- Trace whether frontend actions actually hit real backend logic
- Verify API endpoints exist and behave correctly
- Identify silent failures, missing validation, try/catch gaps, incorrect assumptions, schema mismatches, and bad error handling
- Check whether writes are really persisted and whether data reloads correctly
- Identify any demo-breaking issues related to database seeding, missing records, or environment configuration

5. Data realism / demo setup
- Check whether the current data is believable and usable for a pilot/demo presentation
- Identify missing seed/demo data needed to make the app look complete
- Recommend exactly what sample users, records, dashboards, cases, logs, notes, reports, or scenarios should exist before demo day
- If current demo data is weak, propose a clean demo dataset structure

6. Security / access control for demo
- Not a full enterprise security audit, but identify obvious critical issues
- Especially check:
  - unauthenticated API access
  - cross-user data leakage
  - missing authorization checks
  - hardcoded secrets or dev shortcuts
  - admin functionality exposed to normal users
- Clearly separate:
  - “must fix before demo”
  - “acceptable for demo but not production”

7. Code quality / stability risks
- Identify brittle areas likely to fail during a live walkthrough
- Flag TODOs, mock implementations, commented-out critical logic, inconsistent patterns, duplicated logic, and areas where the app only “looks done”
- Note any areas where refreshing, navigating back, or using slightly unexpected inputs may break things

How to work:
- First, map the app:
  - routes/pages
  - components
  - APIs
  - DB models
  - Clerk integration points
  - key feature modules
- Then test systematically, not randomly
- Prefer breadth first, then depth on important demo flows
- Assume the audience will judge reliability, polish, and credibility
- Assume any broken click during demo is a serious issue

Deliverables:
1. Executive summary
- Overall demo readiness score from 1–10
- Overall pilot readiness score from 1–10
- Top 5 biggest demo risks
- Top 5 fastest wins

2. Feature audit table
For each major feature include:
- Feature name
- Status: Working / Partial / Broken / Placeholder
- Demo risk: Low / Medium / High
- Notes
- Recommended fix

3. Clerk/auth audit
- What is correctly implemented
- What is missing
- What is risky
- Exact routes/endpoints/components needing changes

4. Broken flows list
- Step-by-step reproduction
- Likely cause
- Exact files involved
- Recommended fix approach

5. Demo day checklist
- Exact things I should fix before presenting
- Exact things I should avoid clicking
- Exact demo-safe path through the app
- Exact seed/demo data I should prepare in advance

6. Prioritized remediation plan
Split into:
- Critical before demo
- Important before pilot
- Nice to have later

For each item include:
- title
- why it matters
- severity
- estimated difficulty: S / M / L
- exact files likely involved
- definition of done

Important instructions:
- Be brutally honest and specific
- Do not give vague advice like “improve UX” or “check auth”
- Name exact pages, components, handlers, routes, queries, and failure points
- If something is fake, disconnected, mocked, or not persisted, say so plainly
- If a feature only works under narrow conditions, say that
- If you cannot verify something, explicitly label it “not verified”
- Do not assume a feature works because the UI exists
- Treat this as a real pre-demo audit, not a casual code review

Extra output:
At the end, provide:
A. “If I had only 2 hours before the demo, fix these first”
B. “Safest demo path”
C. “Highest-risk clicks to avoid”
D. “Minimum believable demo dataset”

If possible, also create:
- a punch-list of exact fixes in execution order
- a concise founder-facing summary I can paste back to ChatGPT for implementation help
