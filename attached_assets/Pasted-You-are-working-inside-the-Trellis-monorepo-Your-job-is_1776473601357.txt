You are working inside the Trellis monorepo. Your job is to **reduce product sprawl and refocus the app around the core wedge**:

**Trellis = SPED service-delivery, compliance-risk, and compensatory-exposure management for school districts.**

That means the product should primarily feel like:

* track required vs delivered service minutes
* surface missed sessions and provider coverage gaps
* flag compliance risk early
* show compensatory exposure
* support pilot onboarding, data readiness, and meeting-ready reporting

## Non-negotiable rules

### 1) Do not break anything

* Make this a **safe refactor / feature-hiding pass**, not a destructive rewrite.
* Do **not** delete routes, pages, APIs, DB schema, components, or business logic.
* If something is removed from the user experience, **hide it only**.
* Existing code should remain recoverable and easy to re-enable later.
* Preserve app stability, imports, build health, auth, role gating, and route integrity.

### 2) Hide, don’t delete

If a feature does not support the wedge, do one or more of the following instead of removing it:

* remove from primary nav
* move into a collapsed **“More tools”**, **“Advanced”**, or **“Expanded platform”** area
* keep route accessible but no longer promoted
* keep backend endpoints intact
* keep feature flags / tier gates intact where possible
* if necessary, add a single central config layer for `core`, `secondary`, and `hidden-from-primary-nav`

### 3) Do not create broken dependencies

Before hiding a feature:

* trace whether any visible page, report, workflow, sidebar item, dashboard card, onboarding step, redirect, or API call depends on it
* if something depends on a hidden feature, either:

  * keep the dependency accessible but unpromoted, or
  * reroute the visible workflow to a wedge-aligned destination
* no broken links
* no empty cards caused by removed nav items
* no dead CTAs
* no imports failing because a component was “cleaned up”

### 4) Preserve reversibility

Every hidden feature should be easy to restore later.

* Prefer centralized visibility control over scattered conditionals.
* Add comments where helpful.
* Do not rewrite core business logic unless needed for stability.
* Keep route files and page files intact.

### 5) Be conservative

If unsure whether something is needed, **keep it accessible but de-emphasized** rather than removing it from view entirely.

## Important engineering behavior

* Think like a careful staff engineer doing a product-focus cleanup.
* Prefer minimal, reversible changes.
* Do not delete anything.
* Do not silently break discoverability for workflows that still matter.
* If a hidden feature is needed by a visible flow, keep it reachable.
* If a route is preserved but nav-hidden, make sure it still has sane breadcrumbs/back links where appropriate.
