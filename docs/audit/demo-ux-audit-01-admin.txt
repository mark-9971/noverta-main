DEMO-UX-AUDIT-01 — Admin Demo-User UX Audit (Noverta)
Code-grounded · audit-only · no edits made.

--------------------------------------------------------------------------------
1. EXECUTIVE VERDICT
--------------------------------------------------------------------------------

Admin demo takeover is conditionally safe — but only on a narrow,
pre-scripted path. A skeptical district admin who clicks freely from the
landing page will hit at least four credibility-damaging surfaces within
~90 seconds.

Strongest:
The wedge spine is real and visibly honest. Specifically: shared handling
state (`use-handling-state` writing to `action_item_handling`),
school-calendar-aware minute math (`minuteCalc` weighting closures `0`,
early release `0.5`), the deterministic risk ladder (completed → on_track →
slightly_behind → at_risk → out_of_compliance → no_data), the "Likely
cause → Owner → Recommended action" chain, the `awaiting data`
denominator-honesty banner, and the surprisingly disciplined Medicaid
disclaimers ("Noverta does not file claims with Medicaid. It prepares the
data and tracks it through your internal review."). The "T06 truthful
rename" of "Follow up with provider" → "Mark as awaiting provider
response" shows the team has already started this work — but only on
Action Center.

Riskiest:
(1) The first-load state I observed live: a sample-data banner reading
    "Your workspace includes 2000 sample students and 354 sample staff"
    with the Sample Data Tour modal automatically blocking the dashboard
    while the dashboard itself shows `Loading…` — every one of those four
    signals (huge fake number, mandatory tour, persona switcher,
    indefinite loading) reads as internal QA tool not production buyer
    demo.
(2) Action Center shows "99+" unread — a value that simultaneously
    screams "overwhelm" and "fake demo seed."
(3) The sidebar's `DEMO ROLE` block with four hot-swappable persona
    buttons (Admin / Case Manager / BCBA / Para) is debug scaffolding
    leaking into the buyer surface.
(4) The `More` collapsible nav contains 7 product domains (IEP &
    Documents, Scheduling, People, Clinical, Finance, Settings)
    totaling ~25 routes — many of which the platform-breadth audit
    classified as scaffold or theatrical (ABA Hub, Caseload Balancing,
    Cost Avoidance, Resource Management).

What should be avoided:
ABA / Behavior, Caseload Balancing, Cost Avoidance, Resource Management,
IEP Suggestions ("Programs"), Supervision, the `/scheduling?tab=coverage`
and `?tab=calendar` sub-tabs, and any platform-admin Demo Control Center
surface (Hero Cast, Highlight Mode, Generate Cast — these are operator
tools, not buyer surfaces).

Must fix before a serious buyer demo:
1. Default sample roster from 2000 to ~140 for first-load demo districts.
2. Suppress or auto-dismiss the Sample Data Tour after first viewing per
   browser; never show it on a fresh tab during a live demo.
3. Hide the `DEMO ROLE` persona switcher unless `?devPersona=1` or
   `isPlatformAdmin`.
4. Cap the Action Center badge at `99+` only when actually >99; for a
   140-student demo the real count should be 12-25.
5. Move at least 4 of the 7 `More` groups behind a `Preview` badge or
   `?more=1`.

3. TOP 15 STRENGTHS TO LEAN INTO (RANKED)
--------------------------------------------------------------------------------

 1. Shared handling state with full audit trail. `HandlingHistoryPopover`
    shows from→to transitions, who, when, with notes; states are real
    (`needs_action`, `awaiting_confirmation`, `recovery_scheduled`,
    `handed_off`, `under_review`, `resolved`). Demo this with two browser
    windows showing the same item update across roles.

 2. The truthful rename pattern (T06). "Follow up with provider" → "Mark
    as awaiting provider response". This is the right pattern; buyers
    respect honest labels.

 3. Risk → Likely Cause → Owner → Recommended Action chain. Show the same
    student row driving four different action labels depending on signal
    type (`documentation_lag` → "Confirm & log session";
    `likely_missed_service` → "Schedule makeup"; `schedule_mismatch` →
    "Review with case manager").

 4. School-calendar-aware minute math. Closures weight `0`, early release
    `0.5`. Show a "Behind / On Pace" pill changing based on calendar
    exceptions — competitors do not do this.

 5. `stillAtRiskMinutes` = max(0, Required − Delivered − ScheduledPending).
    The decomposition into `delivered` + `scheduledPending` +
    `stillAtRisk` is the single most defensible honesty story. Lead with
    this in any compliance demo.

 6. `awaiting data` banner. Tells the buyer: "X students excluded from
    health % to prevent fake green status." This is the kind of statement
    a superintendent will repeat to their board.

 7. Quick Log "smart landing" via `computeInitialStep`. Open from a
    session block → land on Outcome step, not Student step. Demo this
    from a row in Action Center.

 8. Quick Repeat / Recent Combos in Quick Log. Actual paraprofessional-
    grade UX (1-tap combo). Most platforms force a multi-screen flow.

 9. Medicaid "data prep, not filing" honesty. "Noverta does not file
    claims with Medicaid. It prepares the data and tracks it through your
    internal review." Direct, defensible, and disarms the obvious
    objection.

10. /weekly-compliance-summary. Mature print/PDF/CSV artifact backed by
    real Minutes-at-Risk data. Bring a printed copy.

11. "Risk priced as comp-ed exposure" framing. Translating minutes
    shortfall to projected dollars is the #1 board-meeting hook.

12. /cm-review/:itemId handoff page. Real handoff with a structured note
    attaches when an admin says "Review with case manager." Demo
    cross-role collaboration here.

13. Snooze with auto-restore + visible restore-by timestamp ("Snoozed 3
    days · by Maria Chen · 2d left") — proves the system never silently
    drops work.

14. Recommended Next Step on Student Detail. Same engine as Action
    Center, contextualized to one student. Demo "what should I do for
    Marcus?" — real answer, not a generic dashboard.

15. CSV exports on every report (Risk Report, Compensatory, Medicaid).
    Districts trust what they can pull into Excel; never hide the export
    button.

4. SAFE DEMO PATH (CLICK-BY-CLICK)
--------------------------------------------------------------------------------

PRE-FLIGHT:
Run /admin/demo-readiness and confirm checks pass; reseed with
targetStudents=140 (Medium); dismiss any active tour; close the persona-
switcher panel; sign in fresh as admin with cookies cleared.

STEP 1 — Route: /
  Say:    "This is what your team sees Monday morning. The number that
           matters is the still-at-risk minutes — not the headline
           compliance %."
  Show:   The dashboard's risk header + handling-state pills.
  Don't:  "It calculates everything itself" — instead: "It tells you what's
           true based on what was actually logged."
  Next:   Action Center.

STEP 2 — Route: /action-center (Urgent tab)
  Say:    "One queue. Whatever's off-track today across schools, providers,
           or students lands here."
  Show:   The Likely Cause + Owner + Recommended Action triple on one row.
  Don't:  "It auto-fixes" or "AI handles".
  Next:   Click `Schedule makeup` on a `likely_missed_service` row.

STEP 3 — Route: /scheduling?tab=minutes (deep-linked from CTA)
  Say:    "It hands you to the minutes-at-risk view, pre-filtered to the
           makeup you're trying to plan."
  Show:   The makeup intent persisting; handling state changes to
           `recovery_scheduled`.
  Don't:  "It auto-schedules" — instead: "You confirm the slot; the
           original alert auto-resolves when you log the session."
  Next:   Back to Action Center; click `Mark as awaiting provider response`
           on another row.

STEP 4 — Route: (same)
  Say:    "One status, every role sees it."
  Show:   Open same item in second window logged in as the case manager —
           pill matches.
  Don't:  Anything about cross-district sync.
  Next:   Student Detail of any at-risk student.

STEP 5 — Route: /students/:id (Summary)
  Say:    "When a parent calls about Marcus, you have the whole picture in
           one place — the next step is named, not implied."
  Show:   Recommended Next Step card; risk badge; metrics; Recent Sessions.
  Don't:  Open the IEP & Goals tab unless asked — content depth varies.
  Next:   Open Quick Log from `Log Session` button.

STEP 6 — Quick Log sheet
  Say:    "30-second logging. It remembers what para Maria usually does
           for Marcus."
  Show:   Quick Repeat tile; smart landing on Outcome.
  Don't:  "Voice / OCR / camera" — not in product.
  Next:   Save → success page.

STEP 7 — Route: /compliance (Risk Report tab)
  Say:    "This is the report a SPED director takes to the school board."
  Show:   "Needs Attention This Period"; Provider Delivery Summary; the
           `awaiting data` banner.
  Don't:  Open the Trends tab on demo data — patterns may look noisy.
  Next:   Print Report or Download CSV.

STEP 8 — Route: /medicaid-billing (only if a billing question is asked)
  Say:    "Noverta prepares the claim drafts; your billing system or
           clearinghouse files them."
  Show:   The export disclaimers and the "Exported" status definition.
  Don't:  "We handle reimbursement".
  Next:   Stop.

TOTAL CLICKS: ~14. HARD STOP after step 8. Do not free-navigate.

20. FINAL RECOMMENDATION
--------------------------------------------------------------------------------

Single highest-value UX move:
Reduce the default demo seed from 2000 to ~140 students AND gate the
Sample Data Tour behind an explicit Start button — both in the
sample-data banner. These two changes alone remove the first three
trust-damaging signals a buyer sees in the first 30 seconds (absurd
roster size, intrusive tour, "99+" badge that's only fake because the
seed is huge), without touching any wedge code or any product capability.
Total effort: less than a day.

Single highest-risk visible issue:
The "Schedule makeup" CTA. Today it routes to /scheduling?tab=minutes
and marks the item recovery_scheduled but does not actually commit a
scheduled session — closure happens only when Quick Log fires. A
skeptical buyer will click the button expecting "schedule" semantics,
see what looks like a deflection, and leave the demo with the impression
that the closed loop is theater. Either rename the CTA to "Plan makeup
recovery" (XS effort, immediate honesty) or build the inline "Confirm
scheduled" microform (M effort, true closed loop).
